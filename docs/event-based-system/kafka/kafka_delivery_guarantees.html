When a producer sends a message, it can ask Kafka: "how many brokers need to confirm before you tell me it's safe?" That's acks. Three options — pick your trade-off between speed and safety.

acks=all Leader + all in-sync replicas confirm. slowest safest Orders, payments — data loss is unacceptable

A retry happens automatically when the broker doesn't respond in time. Without idempotency, the broker may have written the message before crashing — so the retry produces a duplicate.
Idempotency means sending the same message twice has the same result as sending it once. Kafka assigns each message a sequence number. If a duplicate arrives, the broker recognises it and silently drops it.

Enabling idempotency is free — it adds a tiny sequence number to each message. Always enable it. It activates automatically when you set acks=all and retries > 0.
These are the production-safe producer settings for critical data like orders and payments. Each config exists to close a specific failure gap.
// Reliability
props.put("acks", "all"); // wait for all ISR replicas
props.put("retries", Integer.MAX_VALUE); // retry forever
props.put("enable.idempotence", true); // no duplicates on retry
props.put("max.in.flight.requests.per.connection", 5); // safe with idempotence

// Throughput
props.put("linger.ms", 5); // wait 5ms to batch messages
props.put("batch.size", 16384); // 16KB batch max
props.put("compression.type", "snappy"); // compress batches
What each config protects against
ConfigFailure it prevents
acks=allLeader crashes after write but before replication — data gone
retries=MAXTransient network failure drops the message permanently
enable.idempotenceRetry sends duplicate — consumer processes an order twice
linger.ms=5Tiny batches waste network overhead on high-volume topics
compression.typeLarge payloads slow throughput and increase broker disk usage
One shortcut: setting enable.idempotence=true automatically forces acks=all and max.in.flight=5 — you get the safe defaults for free.
