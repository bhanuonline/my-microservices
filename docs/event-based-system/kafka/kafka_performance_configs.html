Kafka: performance configs

Key producer and consumer configs — what they do, when to tune them, and how they trade off against each other.

Throughput configs
linger.ms
default: 0 · recommended: 5–20
How long the producer waits before sending a batch. At 0, every message is sent immediately — tiny batches, high overhead. At 20ms, messages accumulate into larger batches. Higher = more throughput, slightly more latency.
high impact
batch.size
default: 16384 (16KB) · recommended: 32768–131072
Maximum bytes accumulated in a batch before sending. Larger batches = fewer network round trips = more throughput. Works together with linger.ms — whichever limit is hit first triggers the send.
high impact
compression.type
default: none · recommended: snappy or lz4
Compresses batches before sending. snappy = fast compression, good ratio. lz4 = fastest, slightly lower ratio. gzip = best ratio, slowest. Reduces network and disk usage significantly on text/JSON payloads.
high impact
buffer.memory
default: 33554432 (32MB) · recommended: 67108864 (64MB)
Total memory the producer uses to buffer records waiting to be sent. If your producer generates messages faster than the broker can accept, this buffer absorbs the burst. If it fills up, send() blocks.
medium impact
Latency configs
max.in.flight.requests.per.connection
default: 5 · keep at 5 with idempotence
How many unacknowledged sends can be in flight at once. Higher = more parallelism = more throughput. With enable.idempotence=true, 5 is safe and maintains ordering per partition.
trade-off
request.timeout.ms
default: 30000 (30s) · tune based on SLA
How long the producer waits for a broker response before considering a request failed. Reduce for fast-fail behaviour; increase for busy brokers under load.
medium impact
Throughput configs
fetch.min.bytes
default: 1 · recommended: 1024–65536
Minimum bytes the broker accumulates before responding to a fetch request. At 1, the broker responds immediately even if only 1 byte is ready. Higher values mean fewer, larger fetches — better throughput at the cost of slightly more latency.
high impact
fetch.max.wait.ms
default: 500ms · tune with fetch.min.bytes
Maximum time the broker waits before responding even if fetch.min.bytes hasn't been reached. Acts as a safety valve — even a quiet topic will respond within this window. Increase to reduce fetch frequency.
medium impact
max.poll.records
default: 500 · tune based on processing time
Max records returned per poll() call. Higher = more records per batch = better throughput. But if processing one batch takes longer than max.poll.interval.ms, Kafka thinks the consumer is dead and triggers a rebalance. Set this based on how fast your processOrder() runs.
high impact
fetch.max.bytes
default: 52428800 (50MB)
Maximum data returned in a single fetch response. Increase if your messages are large or you want to fill poll() batches more aggressively. Must be larger than max.partition.fetch.bytes.
medium impact
Stability configs
max.poll.interval.ms
default: 300000 (5min) · reduce if possible
Maximum time between poll() calls before Kafka considers the consumer dead and triggers a rebalance. If your processing logic is slow (DB writes, HTTP calls), increase this. Better fix: reduce max.poll.records so each batch finishes faster.
trade-off
heartbeat.interval.ms
default: 3000 · must be < session.timeout.ms / 3
How often the consumer sends a heartbeat to the broker to prove it's alive. Lower = faster failure detection = faster rebalance when a consumer dies. Keep at 3000 unless you have a specific reason to change it.
medium impact
session.timeout.ms
default: 45000 (45s) · recommended: 30000–60000
How long the broker waits without a heartbeat before declaring the consumer dead. Shorter = faster failover. Longer = more tolerance for GC pauses and slow processing. Must be within broker's group.min/max.session.timeout.ms range.
trade-off
